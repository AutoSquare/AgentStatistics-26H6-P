# AgentStatistics 开发说明

项目概述见 [README.md](../README.md)。

持久化路线：**memory**（`memory` = 记忆快照；`archive` = ZIP 工程包）

## 目录职责

| 路径 | 职责 |
|------|------|
| `Themes/` | DefaultTheme 色板与全局样式 |
| `Resources/` | 图标/背景/字体（ASCII 路径） |
| `Properties/` | Settings.settings / Resource.resx |
| `ViewModel/`、`Services/`、`Model/` | MVVM + Python 桥 |
| `ASPy/`、`ASEnv/` | Python 脚本与运行时 |

## Codex 统计链路

- Codex 会话定位、解析、缓存、聚合、额度、费用与 CSV 字段语义以 `../CodexScope/generate_codex_data.go` 和 `../CodexScope/app.ts` 为行为真相源。
- 第一阶段不再使用 Go；`ASPy/codex_usage_stats.py` 迁移 CodexScope 核心逻辑并输出前端 JSON。
- 共享聚合逻辑位于 `ASPy/usage_common.py`；Codex / Cursor / Antigravity 均输出 `schemaVersion: 2` 的同款 JSON，前端复用 `WebClient/src/components/AgentDashboard.vue` 与 `charts.ts`。
- 默认会话目录为 `%USERPROFILE%/.codex/sessions`，用户可在界面修改，保存键为 `codexSessionsPath`。
- 增量缓存写入 `%AppData%/AgentStatistics/codex_usage_cache.json`，不写入仓库。
- WPF 主窗口通过 WebView2 加载 `WebClient/dist/index.html`，并通过 WebView message 发送设置、刷新状态与 Codex 统计结果。
- **多 Agent 扫描隔离**：`MainWindow.xaml.cs` 为 codex / cursor / antigravity 各维护独立 `CancellationTokenSource` 与 `_runningRefresh` 标记；不同 Agent 的刷新可并行，仅同一 Agent 的重复刷新会合并排队。禁止再用全局 `_scanCts` 取消其他 Agent 的 Python 子进程。
- 自动刷新由 WPF 监听 sessions 目录下 `*.jsonl` 文件变化触发；手动刷新按钮保留。
- 隐私边界：只处理 Token、模型、时间戳、会话 id、cwd basename、rate limit、耗时、失败状态等元数据；不读取或导出提示词、助手正文、工具输出或文件内容。

## Cursor 统计链路

- 行为参考 `../token-monitor`：`token-monitor` 通过 `tokscale cursor sync` 维护 `%USERPROFILE%/.config/tokscale/cursor-cache/usage*.csv`；本仓库用 `ASPy/cursor_tokscale_cli.py` 调用同一 CLI（优先 `token-monitor/node_modules` 内置二进制，否则 `npx tokscale@latest`），CSV 解析仍用自研 `ASPy/tokscale_csv.py`。
- 统计入口：`ASPy/cursor_usage_stats.py`；C# 包装：`Services/CursorUsageService.cs`。
- 默认缓存目录 `%USERPROFILE%/.config/tokscale/cursor-cache`，界面可改，保存键为 `cursorCachePath`。
- 登录态来源：只读取 Cursor CLI 的 `~/.cursor/cli-config.json`（账号身份）与 `%APPDATA%/Cursor/auth.json`（access/refresh token）。两者的 `authInfo.authId` 与 JWT `sub` 必须匹配，否则拒绝使用。Cursor 模块不读取桌面 IDE `state.vscdb`、浏览器 Cookie、Edge 或手工保存的 Session Token。
- **用量 API 路由**：CLI 登录态有效时，WPF 直接调用 Python 同步链，使用 CLI Token 拉取 Dashboard events 与 `usage-summary`，写入 `usage.json` 和额度缓存。Cursor 模块没有 Dashboard WebView、网页登录或外部浏览器回退。
- 账号隔离：完整官网快照写入 `usage.json` 时附带稳定 `accountId` 与可用邮箱；`ASPy/cursor_usage_stats.py` 将其归档到 `cursor-cache/accounts/<account-hash>/`。账号目录内保存 `account.json`、`usage.json` 和该账号的历史 CSV。旧版根目录无账号 `usage.csv` 首次迁移时直接归当前登录账号，后续账号不复用其他账号的 CSV。
- 数据源选择：`ASPy/tokscale_csv.py` 在**同一账号内**读取 `usage.json` 与 `usage*.csv`。时间戳和模型相同的记录视为同一事件：官网非零记录覆盖本地记录；官网记录为零而本地存在非零 Token 时保留本地；官网缺失的 CSV 长历史继续保留。不同账号之间不去重，全部账号总计为各账号结果之和。
- 当前账号识别：以 `cli-config.json` 的 `authInfo.authId` / `email` 为当前账号唯一真相源。最新 `usage.json.accountId` 决定该批用量的归属，旧缓存不得覆盖 CLI 当前账号。
- 账号唯一性与在线状态：账号主键统一规范化为 `user_*`，例如 `github|user_*` 与 `user_*` 必须合并为同一账号。一次只能有一个 CLI 当前账号；其他账号仅展示已归档历史。
- Token 口径：Dashboard JSON 的 `tokenUsage.inputTokens` 是未缓存输入，标准化时必须加上 `cacheReadTokens` 与 `cacheWriteTokens`；`cached_input_tokens` 是输入子集，`total_tokens = input_tokens + output_tokens`。该口径与官网 CSV 的 `Total Tokens` 对齐。
- 费用口径：Cursor Dashboard JSON 事件若带 `chargedCents` / `tokenUsage.totalCents`，统一视为官网等效费用并覆盖本地模型价格估算；缺失官网费用的 CSV 历史事件继续按 `pricingRules` 估算。费用结构图按本地估算分项比例缩放到官网总额，确保 KPI、趋势、模型排行与导出 CSV 的费用一致。
- **首次使用同步**：用户必须先在 Cursor CLI 完成登录。AgentStatistics 不提供 Cursor 登录入口；CLI 未登录时仅展示本地缓存与明确状态提示。
- HTTP 请求：`ASPy/cursor_http.py` 先用精简请求头访问 Web API，再回退完整浏览器头与可选 `curl_cffi`；识别 Vercel 拦截后不再提示手填 Cookie。
- CLI API 回退：`ASPy/cursor_cli_api.py` 在 Web API 被 Vercel 拦截时，用 CLI `accessToken` 以 Bearer 访问 `https://api2.cursor.sh/auth/*`。
- 云端同步：`ASPy/cursor_sync.py` 仅使用 CLI Token 调用 Dashboard JSON API；手动刷新和启动同步使用 `--force-sync`，30 秒定时同步受 5 分钟用量快照新鲜度保护，但额度 `usage-summary` 每次实时探测。网络失败时保留上一次完整快照，不以部分数据覆盖。
- **离线保留约束**：未检测到 CLI 登录态、断网、401/403、超时或同步返回不完整时，只能读取现有根缓存与 `accounts/<account-hash>/` 账号归档，不得删除、清空或用空结果覆盖本地记录。只有再次取得有效 CLI 登录态并成功完成该账号的完整同步后，才允许替换该账号的当前快照；其他账号历史仍继续保留。
- 同步节流：`usage.json` / `usage.csv` 在 5 分钟内已更新时，**未带** `--force-sync` 的 `--sync` 会跳过云端拉取（`engine=cache-fresh`）。30 秒定时器、手动刷新与启动首次同步均带 `--force-sync`，保证周期性真同步。
- 额度风险：`ASPy/cursor_limits.py` 优先使用 CLI token 调用 `usage-summary`，可直接获得 Pro+ 的 Auto/API 百分比、金额和周期结束时间；失败时回退 CLI/IDE Bearer API，再回退本地额度缓存。用户手动刷新、启动同步和 30 秒定时同步均执行实时探测，不因 5 分钟缓存而跳过请求。
- 增量解析缓存：`%AppData%/AgentStatistics/cursor_usage_cache.json`。
- 自动刷新：WPF 监听 `cursor-cache` 下 `usage.json` / `usage*.csv` 变化（仅本地快读）；另每 **30 秒**使用 CLI Token 尝试一次联网同步，手动刷新同样强制同步。同步失败时继续展示本地缓存。
- CLI 切号监听：WPF 同时监听 `~/.cursor/cli-config.json` 与 `%APPDATA%/Cursor/auth.json`。切换账号时两文件可能先后写入；任一文件变化都会触发一次强制同步。状态尚未一致时只显示并保留本地历史，不回退 WebView 官网登录；两文件收敛后下一次文件事件立即使用新账号同步，不受旧 WebView 退避时间影响。
- **与 Codex 的差异**：Codex 在 `~/.codex/sessions/*.jsonl` 边对话边落盘；Cursor CLI 的完整计费用量与额度来自 Cursor 服务端。因此离线只能读取最近缓存，无法刷新完整用量或额度。
- 额度缓存：成功探测 `usage-summary` 后写入 `%AppData%/AgentStatistics/cursor_limits_cache.json`；API 失败或暂无 Token 时回退缓存并在风险面板标注「缓存额度」。
- WebView 消息：`cursorData`、`refreshCursor`、`setCursorCachePath`。
- UI 简化：Cursor 页顶部不展示 `cursor-cache` 路径配置；默认使用 `%USERPROFILE%/.config/tokscale/cursor-cache`，用户只需要关注登录状态与刷新按钮。额度面板只显示简短标题，不展示 `usage-summary API` 等内部来源文案。
- 多账号 UI：Cursor payload 顶层 `accounts` 提供每个账号的 `views`、`records`、邮箱、短 ID、当前账号和同步状态。时间范围下方使用固定“全部账号”总览卡 + 横向滚动账号胶囊条；Cursor CLI 当前账号显示“当前”并排在账号列表第一位。
- 空数据语义：payload 顶层 `dataStatus`、`sync`、`auth`；前端无 `records` 时显示空状态而非全 0 图表。

### Cursor 故障排查

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 提示「未检测到 Cursor CLI 登录态」 | `cli-config.json` 缺少账号，或 `auth.json` Token 的 JWT `sub` 与账号不匹配 | 先在 Cursor CLI 完成登录，再回到 AgentStatistics 刷新 |
| 风险面板仅显示 IDE API / 缓存额度 | `cursor.com` 被 Vercel 拦截，已回退 `api2.cursor.sh` 或本地缓存 | 确认本机 Cursor 已登录；稍后重试 Web API；用量 CSV 仍会尽量从本地 `usage.csv` 读取 |
| 状态栏「同步失败 HTTP 401/403」 | 登录缓存或 tokscale 凭证过期；或同上 Vercel 拦截 | 重新登录 Cursor 账号、更新 Session Token，或更换网络后重试 |
| 云端同步失败但 Dashboard 有数据、界面显示 0 | WebView 同步阻塞 UI 线程，或 `/cn/dashboard/` 被误判未就绪 | 新版本先推送本地 `usage.csv` 再后台同步；已修复 locale 路径识别；确认 `cursor-cache/usage.csv` 存在 |
| 崩溃 `调用线程无法访问此对象`（`PostJson` / `DashboardWebView`） | Python 统计在后台线程结束后直接调用 `PostJson` | 已改为 `PostJson` 自动 `Dispatcher.BeginInvoke` 回 UI 线程；Cursor 统计 await 使用 `ConfigureAwait(true)` |
| 风险面板「缓存额度 · API 暂不可用」 | Cursor API 暂不可用 | 保留最近一次 CLI 额度缓存；检查网络与 Cursor CLI 登录态后刷新 |
| 用量不再实时更新 | 云端同步被拦截、分页未拿全，或本地 CSV 未刷新 | 确认 `cursor-cache/usage.json` 与 `usage.csv` 的 `LastWriteTime`；新版本会合并两者，修复 Token / 网络后等待 30 秒定时同步或手动刷新 |
| 额度正常但状态栏仍写「云端同步暂时受阻」 | 多走 IDE API 额度；用量 CSV 仍用旧版 Debug 输出（缺 `ASPy/cursor_tokscale_cli.py` 与 `ThirdParty/tokscale/win-x64/tokscale.exe`） | 重新编译（`ASPy` 已改为 `Always` 复制）；确认 exe 同目录存在上述文件后手动刷新 |
| 状态栏「同步超时」或 Python `TimeoutError` | Cursor 导出 CSV 响应慢或网络不稳定 | 稍后重试；若 `cursor-cache/usage.csv` 已存在，刷新仍会读取本地缓存 |
| `cursor-cache` 无 `usage*.csv` | 同步未成功或路径错误 | 检查 `%USERPROFILE%\.config\tokscale\cursor-cache`；确认网络可用 |
| CSV 有文件但仍无数据 | 仅表头无数据行，或日期超出默认 30 天窗口 | 确认近期有 AI 调用；切换「历史」视图 |
| 已用 token-monitor 仍无数据 | tokscale 凭证无效或 cursor-cache 为空 | 在 token-monitor 设置中重新登录 Cursor 账号 |

## Antigravity 统计链路

- **CLI 优先**：不依赖 Antigravity IDE。数据根目录优先 `~/.gemini/antigravity-cli`，兼容 `antigravity` / `antigravity-ide` 旧路径（`ASPy/antigravity_paths.py`）。
- 刷新同步：从运行中的 **Antigravity CLI（agy）** 内嵌 `language_server` 经 Connect RPC 拉取轨迹元数据，写入 `antigravity-cache/sessions/*.jsonl` 与 `manifest.json`（`ASPy/antigravity_sync.py`）。进程发现支持 `agy.exe`、CLI 日志端口与无 CSRF 探测（`ASPy/antigravity_connect.py`）。
- Token 历史（离线）：① `antigravity-cache/sessions/*.jsonl`（`ASPy/antigravity_jsonl.py`）；② `brain/<id>/.system_generated/logs/transcript*.jsonl`（`ASPy/antigravity_transcript.py`）。两者在 `antigravity_usage_stats.py` 合并。
- 统计入口：`ASPy/antigravity_usage_stats.py`（支持 `--sync`）；C# 包装：`Services/AntigravityUsageService.cs`（刷新默认带 `--sync`）。
- 默认缓存目录 `%USERPROFILE%/.config/tokscale/antigravity-cache`，保存键为 `antigravityCachePath`；WPF 监听 `sessions/*.jsonl` 与 `~/.gemini/antigravity*/brain/**/*.jsonl` 变化自动刷新。
- 自动触发：Codex 由自身 sessions 文件变化触发；Antigravity 的 Connect RPC 不会主动写入 `antigravity-cache`，因此 WPF 额外用 15 秒定时器执行一次 Antigravity `--sync`，避免只能手动刷新。
- 刷新分层：文件监听（`sessions/*.jsonl`、transcript）仅触发**本地快读**（`sync=false`，约 300ms 防抖）；15 秒定时器、手动刷新、启动后第二次刷新才执行 `--sync`。启动时先快读本地历史，再在后台同步 CLI。
- 连接发现：`detect_connections()` 在单次 Python 调用内复用 12 秒 TTL 缓存；`build_payload()` 只探测一次并传给 sync / quota，避免重复 PowerShell + 端口探测。
- 配额风险：`ASPy/antigravity_quota.py` 复用 `antigravity_connect` 调用 `GetUserStatus`（失败时回退 `GetCommandModelConfigs`）；模型配额按 token-monitor 口径折叠为 **Gemini Pro / Gemini Flash / Claude** 三池，并过滤 placeholder 模型 ID。成功探测后写入 `%AppData%/AgentStatistics/antigravity_quota_cache.json`；CLI 未运行时优先回退该缓存并在风险面板标注「缓存额度」，仅从未成功探测过才显示等待状态。
- 离线读取约束：刷新时即使 `--sync` 因 Antigravity CLI 未运行失败，也必须继续读取本地 `antigravity-cache/sessions/*.jsonl` 与 CLI transcript；只要本地解析到 records，`dataStatus` 必须为 `ok`，WPF 状态栏显示“本地历史用量”而不是空数据。
- **离线清理约束**：CLI 未运行或 Connect RPC 失败时，不得改写 `manifest.json`，也不得删除 `sessions/*.jsonl`。旧会话清理只能发生在存在有效连接并完成新 manifest 写入的同步流程中。
- 增量解析缓存：`%AppData%/AgentStatistics/antigravity_usage_cache.json`。
- WebView 消息：`antigravityData`、`refreshAntigravity`、`setAntigravityCachePath`。
- 同步容错：`antigravity_sync.py` 写入 session JSONL 后必须尽快保存 `manifest.json`；单个 Connect RPC 断连只能记录到 `sync.error`，不能阻断已成功导出的 session 进入后续普通解析。
- 去重约束：`antigravity_jsonl.py` 同时读取 `manifest.json` 和 `sessions/*.jsonl` 时必须用解析后的绝对路径去重，避免同一 session 既由 manifest 引用又被目录扫描命中后重复计数。
- Connect RPC scheme：`antigravity_connect.py` 探测连接时必须记录实际 `http`/`https` scheme，后续 `rpc_request()` 只使用已探测 scheme；不要对 HTTPS 端口先发 HTTP 请求，否则 Antigravity CLI 终端会刷出 `client sent an HTTP request to an HTTPS server`。
- 配额与缓存口径：配额探测与同步共用同一次 `detect_connections()` 结果；池化时同一池取**剩余额度最低**的模型；Antigravity 的 `cacheReadTokens` 可大于本轮新增输入，因此缓存命中率使用 `cached / (input + cached)`，不能直接用 `cached / input`。
- Token 口径（与 Cursor 对齐）：解析阶段经 `ASPy/antigravity_usage_fields.py` 将原始 `input` + `cacheRead` + `cacheWrite` 折叠为 `input_tokens`（prompt 侧总量），`cached_input_tokens` 为缓存子集；KPI「输入」与趋势图「真实输入」均使用该总量，小字「缓存」为子集。`total_tokens = input_tokens + output`；费用与缓存命中率复用 `usage_common.py` 默认重叠口径（与 Codex/Cursor 一致）。

### Antigravity 故障排查

| 现象 | 常见原因 | 处理 |
| --- | --- | --- |
| 全 0 / 空状态 | CLI 未运行且本地无 JSONL / transcript | 运行 `agy` 发起对话，等待自动同步或手动刷新 |
| 状态栏「CLI 未运行」 | 无 `agy` 进程且 CLI 日志端口不可达 | 在终端运行 `agy` 并保持会话 |
| 同步成功但无记录 | 近期无对话，或 RPC 未返回 `generatorMetadata` | 在 CLI 内发起一次对话后重试 |
| 配额面板空 | Connect RPC 不可达且从未成功缓存过额度 | 至少运行一次 `agy` 并完成刷新；之后 CLI 关闭仍可读 `antigravity_quota_cache.json` |

## 跨 Agent 总计页

- 导航 id 为 `total`，前端在 `WebClient/src/App.vue` 挂载与单 Agent 相同的 `AgentDashboard.vue`。
- **无独立 Python / WPF 管线**：不新增 `total_usage_stats.py`、`TotalUsageService` 或 `totalData` WebView 消息；启动时宿主仍会分别推送 `codexData`、`cursorData`、`antigravityData`。
- 聚合逻辑位于 `WebClient/src/aggregatePayload.ts` 的 `mergeAgentPayloads()`：将三份 `AgentPayload` 合并为 `source: "total"` 的虚拟 payload。
- **records 合并**：拼接三源 `records`，`sid` 加 `codex:` / `cursor:` / `antigravity:` 前缀防碰撞；Cursor 使用顶层 `records`（已由 `combine_cursor_accounts` 合并全账号），勿再遍历 `accounts` 避免重复计数。
- **views 合并**：对 `today` / `24h` / `7` / `30` / `history` 的 `summary`、`cost`、`trend`、`distribution` 按时间桶对齐求和；会话与模型排行合并后重排取 Top N。
- **费用**：各源估算（含 Cursor 官网 `chargedCents` 已体现在单源 payload 中）相加；`pricingRules` 按 `label` 去重合并供 CSV 导出估算。
- **风险面板**：总计页 `risk` 置空，`AgentDashboard` 在无风险行时隐藏「额度与风险」面板；额度请查看各 Agent 独立页。
- **刷新**：总计顶栏「刷新全部」并行发送 `refresh`、`refreshCursor`、`refreshAntigravity`；WPF 侧各 Agent 独立 `CancellationTokenSource`，可并行扫描。
- **状态栏**：`buildTotalStatusKind` / `buildTotalStatusMessage` 汇总三源扫描状态，任一源 `scanning` 时总计页显示扫描中。

## UI/UX 约束

- 任何涉及 WPF 或前端界面结构、组件、图表、导航、空状态、视觉设计或交互改动的任务，先调用 `ui-ux-pro-max`。
- 当前设计系统基线：Data-Dense Dashboard，浅色专业仪表盘，主色 `#1E40AF`，强调色 `#D97706`，图表使用 ECharts。
- 交付前至少检查：文本对比度、焦点态、图表空状态、加载/失败状态、按钮禁用态、375/768/1024/1440 宽度无重叠。
- 图表坐标轴必须启用标签容纳，Token 数值使用 `万`/`亿` 短格式，不能出现坐标轴文字被面板裁掉。
- 历史总览从已有记录开始展示，图表按当前画布宽度自适应聚合为小时、天、月、年；不要把跨天历史图固定显示成 `HH:mm` 小时横轴。
- WPF 主窗口 normal 状态的可用下限为 `1280x820`；启动恢复 `%AppData%/AgentStatistics/user_settings.json` 中的旧窗口宽高时会按 `MinWidth`/`MinHeight` 夹紧，退出保存时同样写入夹紧后的尺寸，避免历史 `1040x680` 等过小布局继续造成 WebView2 内容区被压缩。最大化、还原或尺寸变化后宿主会强制布局刷新，并向前端发送 `dashboardResize`；前端用 `ResizeObserver` 监听主内容区，新增图表时必须接入同一 resize 队列。
- 侧边栏 Tab 使用 `v-if` 仅挂载当前 `AgentDashboard`；切走会卸载组件，切回时新实例 `active` 首帧即为 `true`，非 `immediate` 的 `watch(active)` 不会触发。`AgentDashboard` 须在 `onMounted` 与 `watch(chartView/active, { immediate: true })` 中调用 `tryRenderCharts()`，否则 KPI 有数但 ECharts 空白，需切换时间范围才能恢复。

## 构建

2. `cd WebClient && npm install && npm run build`

**WebClient 改动与 WPF 编译顺序**：宿主只加载 `WebClient/dist/`。修改 `WebClient/src/` 后必须先 `npm run build` 再 `dotnet build` / F5；仅改 `ASPy/` 或 C# 时可跳过前端构建。详见 [README.md](../README.md)「修改 WebClient 后、重新编译 WPF 之前必做」。

## 发布 Python 运行时

- `ASEnv/` 是可移植 Python 环境目录，会进入 Release 输出和安装包。
- 发布版 Python 使用便携 Python 3.10.11，放在 `ASEnv/`，应用会优先解析 `ASEnv/python.exe`。
- 发布前必须确认 `ASEnv/` 和 Release 输出中没有 `pyvenv.cfg`、`C:\Users\25705` 或 `Python310` 这类开发机绝对路径。

## 发布 WebView2 运行时

- `ThirdParty/WebView2/MicrosoftEdgeWebView2RuntimeInstallerX64.exe` 是随安装包携带的 Microsoft WebView2 Evergreen Standalone Installer。
- `AgentStatistics安装包/安装包.iss` 会检查 WebView2 Runtime 注册表版本；目标机缺失时自动以 `/silent /install` 静默安装。
- 应用运行时不要求最终用户手动安装 WebView2；如果安装包未能准备运行时，只提示重新运行完整安装包或联系维护人员。

## 版本与安装包

- 软件 EXE 版本在 `AgentStatistics.csproj` 的 `Version`、`AssemblyVersion`、`FileVersion`、`InformationalVersion` 中维护。
- Inno Setup 安装包版本在 `AgentStatistics安装包/安装包.iss` 的 `MyAppVersion` 中维护，并通过 `VersionInfoVersion`、`VersionInfoProductVersion` 写入安装包 EXE 文件属性。
- 发版时保持上述版本号一致，先构建 Release 输出，再确认 `ASEnv/python.exe`、`WebClient/dist/index.html` 和 WebView2 离线安装器都存在，最后编译 `.iss` 安装脚本。
